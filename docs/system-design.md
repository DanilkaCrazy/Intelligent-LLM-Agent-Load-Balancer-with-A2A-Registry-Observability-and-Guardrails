# System Design

## Ключевые архитектурные решения
1. **Stateless Proxy:** Платформа не хранит состояние диалогов. Это упрощает масштабирование и снижает риски утечки данных.
2. **In-memory Registry (для PoC):** Реестры агентов и провайдеров хранятся в памяти для скорости разработки. В production заменяется на Redis/PostgreSQL.
3. **Streaming Support:** Поддержка SSE (Server-Sent Events) для потоковой передачи ответов от LLM без буферизации, что критично для UX.

## Список модулей и их роли
- **API Gateway:** Точка входа, терминация SSL, базовая валидация HTTP.
- **Auth Middleware:** Проверка прав доступа агентов.
- **Guardrails:** Фильтрация вредоносных промптов.
- **Balancer / Router:** Ядро системы. Выбирает провайдера на основе метрик (Latency, Health, Priority).
- **Telemetry Collector:** Сбор TTFT, TPOT, Cost, RPS и экспорт в Prometheus.
- **Frontend Dashboard:** React-приложение для визуализации состояния системы.

## Основной workflow
1. Запрос поступает на `/api/v1/chat/completions`.
2. `Auth Middleware` проверяет токен.
3. `Guardrails` проверяет последний промпт на инъекции.
4. `Balancer` ищет доступных провайдеров для запрошенной модели, сортирует по приоритету и задержке.
5. Запрос проксируется к выбранному провайдеру.
6. При получении первого байта фиксируется TTFT.
7. Ответ стримится клиенту, параллельно считаются токены.
8. По завершении обновляется статистика провайдера (Latency, Health) и отправляются метрики.

## State / Memory / Context handling
Платформа **stateless**. Управление контекстом (Memory) полностью лежит на стороне вызывающих агентов. Платформа лишь проксирует массив `messages`.

## Retrieval-контур
В рамках данного PoC платформа выступает как инфраструктурный слой. Retrieval (RAG) выполняется агентами до обращения к платформе. Платформа получает уже аугментированный промпт.

## Tool / API-интеграции
- **LLM APIs:** Интеграция по стандарту OpenAI API (наиболее распространенный контракт).
- **Prometheus:** Интеграция через `/metrics` endpoint (pull-модель).

## Failure modes, fallback и guardrails
- **Провайдер недоступен:** Балансировщик помечает его `unhealthy` и берет следующего из пула.
- **Таймаут провайдера:** Соединение обрывается, клиенту возвращается 504, провайдер пенализируется.
- **Срабатывание Guardrails:** Клиенту немедленно возвращается 403 Forbidden с указанием причины.

## Ограничения
- **Latency:** Overhead платформы < 50ms.
- **Cost:** Платформа сама по себе потребляет минимум ресурсов (Node.js/Express).
- **Reliability:** 99.9% uptime для самого Gateway (достигается горизонтальным масштабированием).
